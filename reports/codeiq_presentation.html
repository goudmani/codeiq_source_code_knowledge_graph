02 Problem & Approach

Developers lose hours navigating code they didn't write.
LLMs alone make it worse.

The problem

  • Structure is invisible. In a ~100K-line React Native codebase, questions like "which hook owns session state?" or "what breaks if I change this component?" take senior-developer time that grep and IDE search can't replace: they match text, not relationships.
  • Onboarding is slow. New engineers spend weeks building the mental map of components, hooks, and screens before they can ship safely.
  • Raw LLMs hallucinate code. Asked directly, a general-purpose model invents plausible file paths and component names. Confident, unverifiable, and unusable for engineering decisions.

Our approach

  • Parse the codebase into a knowledge graph. Every file, component, hook, and screen becomes an entity; renders / calls / depends_on / defines relationships become edges.
  • Add a semantic layer. A ChromaDB vector index with LLM-written entity descriptions makes the graph searchable in plain English.
  • Constrain the LLM to the evidence. A tool-calling agent can only answer through graph and index lookups. Every answer ships with cited sources and a deterministic confidence score.

08 Hallucination & AI Risk Strategy

Four layers between the model and a made-up answer

1
The agent only sees the graph
The LLM cannot answer from its training memory. Its only access to the codebase is through four retrieval tools over the extracted entities and edges. No retrieval, no claim.
2
Deterministic confidence scoring
Confidence is computed from retrieval relevance, never self-reported by the model, so it's reproducible and can't be gamed by a persuasive answer.
● HIGH ● MEDIUM ● LOW
3
Every answer cites its sources
Answers ship with exact file paths and line ranges, rendered from the live source on disk, so any claim is one click away from human verification. Low-confidence answers are visibly flagged in the UI.
4
Continuous evaluation harness
A golden set of 10 questions with expected entities runs against every candidate model (make eval); entity-hit rate, blank answers, errors, and latency are tracked and version-controlled.
src/qa_agent/agent.py · the actual scoring rule deterministic · reproducible 
def _score_confidence(sources):
    top = max(relevance scores of retrieved entities)
    strong_support = count(score >= 0.6)
    if top >= 0.75 and strong_support >= 2:  return "High"   # multiple strong sources
    elif top >= 0.5:                         return "Medium" # plausible, verify cited code
    else:                                    return "Low"    # weak evidence, flagged in UI
