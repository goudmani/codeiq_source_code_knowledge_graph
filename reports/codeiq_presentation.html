03 Problem & Approach

Developers lose hours navigating code they didn't write.
LLMs alone make it worse.

The problem

  • Structure is invisible. Grep and IDE search match text, not relationships.
  • Onboarding is slow. Weeks pass before new engineers can ship safely.
  • Raw LLMs hallucinate code. Invented file paths are useless for engineering decisions.

Our approach

  • Parse the code into a knowledge graph. Files, components, hooks, and screens, linked by real relationships.
  • Add a semantic layer. The graph becomes searchable in plain English.
  • Constrain the LLM to the evidence. Answers come only from the graph and index, with cited sources and a confidence score.

05 System Architecture

Built once per repo, queried live on every question

Repo-agnostic by design: every stage's output is namespaced under a <owner>_<repo>_<branch> tag, so pointing the pipeline at a second codebase never touches the first. Each stage is independently re-runnable.

  • Faster onboarding: new engineers query the codebase's structure instead of reverse-engineering it for weeks.

  • Fewer interruptions: routine "where / what / why" questions stop landing on senior developers.

08 Cost Profile

Where every token goes

Read: 106 logged LLM calls across the 30-question eval, roughly 14,300 tokens per question on openai/gpt-oss-120b. The system prompt alone accounts for 57.7% of all tokens and search_code results another 30.8%, so prompt trimming and retrieval caching are the two biggest cost levers. Identifier-lookup questions run hottest at 16.4K tokens on average.
09 Hallucination & AI Risk Strategy

Four layers between the model and a made-up answer

1
The agent only sees the graph and index
The LLM cannot answer from its training memory. Its only access to the codebase is through four retrieval tools over the extracted entities and edges. No retrieval, no claim.
2
Deterministic confidence scoring
Confidence is computed from retrieval relevance, never self-reported by the model, so it's reproducible and can't be gamed by a persuasive answer.
● HIGH ● MEDIUM ● LOW
3
Every answer cites its sources
Answers ship with exact file paths and line ranges, rendered from the live source on disk, so any claim is one click away from human verification. Low-confidence answers are visibly flagged in the UI.
4
Continuous evaluation harness
A golden set of 10 questions with expected entities runs against every candidate model (make eval); entity-hit rate, blank answers, errors, and latency are tracked and version-controlled.
